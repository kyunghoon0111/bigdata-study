## 이번 세션에서 생성할 범위
- 과목: {과목명}
- 파트: {파트명}
- 세부항목 및 세세항목:

## 이번 세션 생성 수량
- 개념 카드: 최소 {X}개 이상
- 퀴즈: 정확히 {Y}개

## JSON 스키마

### 개념 (Concept)
```json
{
  "id": "S{과목번호}_P{파트번호}_C{순번3자리}",
  "subject": "과목명",
  "part": "파트명",
  "section": "세부항목명",
  "subtopic": "세세항목명",
  "title": "개념 제목",
  "content": "핵심 설명 (3~5문장, 시험 출제 관점으로 압축)",
  "keywords": ["키워드1", "키워드2"],
  "level": "core 또는 advanced",
  "examTip": "시험에서 자주 나오는 함정/비교/오개념 포인트 (1~2문장)",
  "verificationStatus": "verified 또는 review_needed 또는 law_check",
  "verifiedAt": "{today}",
  "sourceType": "general 또는 law 또는 policy"
}
```

### 퀴즈 (Quiz)
```json
{
  "id": "S{과목번호}_P{파트번호}_Q{순번3자리}",
  "subject": "과목명",
  "part": "파트명",
  "section": "세부항목명",
  "subtopic": "세세항목명",
  "question": "문제 텍스트 (실제 시험 스타일의 4지선다)",
  "options": ["A) ...", "B) ...", "C) ...", "D) ..."],
  "answerIndex": 0,
  "explanation": "왜 정답인지 + 오답이 왜 틀린지 (이해 중심, 2~3문장)",
  "level": "core 또는 advanced",
  "verificationStatus": "verified 또는 review_needed 또는 law_check",
  "relatedConceptIds": ["S1_P1_C001"]
}
```

## 생성 규칙

### 개념 생성 규칙
1. 세세항목(subtopic) 1개당 최소 1개 이상의 개념 카드 생성
2. content는 백과사전식 나열이 아니라 "시험에서 묻는 관점"으로 작성
3. examTip은 반드시 포함 — "상관관계≠인과관계", "n≥30은 경험칙" 같은 함정 포인트
4. level 기준:
   - core: 필기 합격에 필수적인 기본 개념 (전체의 약 80%)
   - advanced: 알면 좋지만 합격에 필수는 아닌 심화 내용 (UMAP, SHAP, GAN 등)
5. verificationStatus 기준:
   - verified: 일반적인 학술/기술 개념으로 변동 가능성 낮음
   - law_check: 법·제도·정책 관련 내용으로 최신성 검증 필요
   - review_needed: 아직 교차 검증이 안 된 내용
6. sourceType 기준:
   - general: 통계, ML, 데이터 분석 등 일반 기술 개념
   - law: 개인정보보호법, 데이터 3법 등 법령 관련
   - policy: 한국 데이터 정책, 마이데이터, 거버넌스 등
7. subtopic 필드에는 출제기준의 세세항목명을 정확히 기입

### 퀴즈 생성 규칙
1. 실제 시험 스타일: "다음 중 ~에 대한 설명으로 가장 적절한 것은?", "다음 중 ~이/가 아닌 것은?"
2. 오답 선지도 그럴듯하게 — 단순히 틀린 게 아니라 헷갈리게 만들기
3. explanation은 "정답 이유"뿐 아니라 "주요 오답이 왜 틀린지"도 간단히 포함
4. relatedConceptIds는 배열로, 해당 문제가 참조하는 개념 ID를 모두 연결
5. answerIndex는 0-based (0=A, 1=B, 2=C, 3=D)
6. 문제 유형 배분:
   - 정의/개념 구분: 30%
   - 비교/차이점: 25%
   - 적용/사례: 20%
   - "아닌 것" 찾기: 15%
   - 순서/절차: 10%

### 통계 표현 주의사항
- "n≥30이면 정규분포" → "n≥30은 자주 쓰이는 경험칙이며 절대 기준은 아님"
- "VIF > 10이면 다중공선성" → "VIF 4 이상 주의, 10 이상 심각"
- "p < 0.05이면 유의" → "유의수준은 사전에 설정하며, 통상 0.05를 사용"

### 법·제도 주의사항
⚠️ 다음 사항은 반드시 공식 자료 검색으로 확인 후 반영하라.
확인되지 않으면 단정하지 말고 verificationStatus: "law_check" 또는 "review_needed"로 표기하라.

확인 필요 항목 (검색 후 반영):
- 개인정보보호법 최신 개정사항 및 시행일
- 가명정보 결합 절차 (결합전문기관, 반출심사)
- 개인정보 전송 요구권 시행 현황
- 최근 개정 사항: 징벌적 과징금 도입, CEO/CPO 책임 강화, ISMS-P 제도 변화 (검색 결과 기준으로 반영)

서술 원칙:
- "데이터 3법"은 2020년 개정의 역사적 배경으로 설명
- 현행 체계는 개인정보보호법 중심으로 서술
- 정보통신망법을 개인정보 보호의 독립 축으로 설명하지 않을 것
- 법·제도 항목은 반드시 verificationStatus: "law_check", sourceType: "law" 설정

## 출력 형식

설명, 제목, 코드펜스, 마크다운 없이 순수 JSON만 출력하라.

생성을 시작하라.
```

## 프롬프트 2: JSON 갭 분석 및 보강 (생성 후 품질 검증용)

```
당신은 빅데이터분석기사 필기시험 학습 콘텐츠 품질 검수자입니다.

아래에 첨부한 JSON 데이터를 검토하고, 다음 기준으로 갭 분석을 수행해주세요.

## 검토 기준

### 1. 커버리지 검사
- 출제기준의 세세항목(subtopic)이 모두 개념 카드에 반영되었는가?
- 빠진 세세항목이 있으면 목록으로 정리

### 2. 정확성 검사
- content 내용에 사실 오류가 있는가?
- 통계 표현이 지나치게 단정적인가? (n≥30, VIF>10 등)
- 법·제도 내용이 현행 기준과 다른가?

### 3. 시험 적합성 검사
- examTip이 실제로 시험에서 의미 있는 함정 포인트인가?
- 퀴즈 난이도가 필기시험 수준에 적절한가?
- 오답 선지가 충분히 매력적인가 (너무 뻔하지 않은가)?

### 4. level 분류 검사
- core로 분류된 항목 중 advanced로 옮겨야 할 것은?
- advanced로 분류된 항목 중 core로 올려야 할 것은?

### 5. 퀴즈-개념 연결 검사
- relatedConceptIds가 정확하게 매핑되어 있는가?
- 개념은 있는데 관련 퀴즈가 없는 항목은?

## 프롬프트 3: HTML 앱 생성

```
당신은 프론트엔드 개발자입니다.
빅데이터분석기사 필기시험 학습 앱을 단일 HTML 파일로 만들어주세요.

## 데이터 구조

앱에서 사용하는 JSON 데이터는 다음 스키마를 따릅니다:

### 개념 (Concept)
- id, subject, part, section, subtopic, title, content, keywords
- level: "core" | "advanced"
- examTip, verificationStatus, verifiedAt, sourceType

### 퀴즈 (Quiz)
- id, subject, part, section, subtopic, question, options
- answerIndex (0-based: 0=A, 1=B, 2=C, 3=D)
- explanation, level, verificationStatus
- relatedConceptIds (배열)

## 시험 정보
- 4과목: 빅데이터 분석 기획 / 빅데이터 탐색 / 빅데이터 모델링 / 빅데이터 결과해석
- 과목당 20문제, 총 80문제, 120분
- 합격: 과목당 40점 이상(8문제) + 전과목 평균 60점 이상

## 앱 기능 요구사항

### 1. 메인 화면
- 4과목 카드 형태로 표시
- 각 과목별 학습 진행률 표시 (파트별 퀴즈 정답률 기반)
- 전체 진행 요약 (총 학습한 개념 수, 총 풀어본 퀴즈 수, 평균 정답률)
- 모의고사 바로가기 버튼

### 2. 개념 학습 모드
- 과목 → 파트 → 세부항목 순으로 네비게이션
- 개념 카드: title, content, keywords, examTip 표시
- level 토글: 기본(core만) / 심화(advanced 포함)
- law_check 항목에는 "⚠️ 최신 법령 확인 필요" 배지 표시
- 카드 넘기기 (이전/다음)

### 3. 파트별 퀴즈 모드
- 해당 파트의 퀴즈를 순서대로 또는 랜덤으로 풀기
- 문제 풀면 즉시 정답/오답 피드백 + explanation 표시
- 관련 개념 보기 버튼 (relatedConceptIds 연결)
- level 필터: core만 / 전체

### 4. 오답노트
- 틀린 문제 자동 저장 (localStorage)
- 과목별/파트별 필터링
- 재풀기 기능
- 오답 횟수 표시
- 오답노트 초기화 버튼

### 5. 모의고사 모드
- 실제 시험과 동일한 구성: 4과목 × 20문제 = 80문제
- 출제 기본 조건: verificationStatus === "verified"인 문제만 출제
- fallback 규칙: verified 문제가 80문제 미만이면 동일 파트의 review_needed를 보조로 사용하되, 결과 화면에 "⚠️ 검증 미완료 문항 N개 포함" 배지를 표시
- 120분 타이머 (시간 초과 시 자동 제출)
- 과목별 랜덤 출제, 파트 비율은 세세항목 수 기반 가중치:
  * 과목1: 빅데이터의 이해 7, 데이터분석 계획 5, 데이터 수집/저장 8
  * 과목2: 데이터 전처리 7, 데이터 탐색 7, 통계기법 이해 6
  * 과목3: 분석모형 설계 6, 분석기법 적용 14
  * 과목4: 분석모형 평가/개선 10, 분석결과 해석/활용 10
- 채점 결과:
  * 과목별 점수 (20문제 중 정답 수 × 5점)
  * 과락 여부 체크 (40점 미만 과목 빨간색 표시)
  * 전체 평균
  * 합격/불합격 판정
  * 오답은 자동으로 오답노트에 추가

### 6. 학습 진행률 대시보드
- 과목별·파트별 정답률: CSS 막대 그래프 (순수 div+CSS, Canvas 미사용)
- 최근 모의고사 점수 이력 (최대 5회): 간단한 표 형태
- 취약 파트 하이라이트 (정답률 60% 미만인 파트 빨간색 표시)

## 기술 요구사항
- 단일 HTML 파일 (CSS, JS 인라인)
- 외부 라이브러리 없음 (순수 HTML/CSS/JS)
- 차트/그래프는 순수 CSS 막대 그래프 또는 HTML 표로 구현 (Canvas/SVG 미사용)
- localStorage로 진행 상태, 오답노트, 모의고사 이력 저장
- 모바일 반응형 (친구가 폰으로도 볼 수 있게)
- 한국어 UI
- 다크모드 지원 (토글)
- JSON 데이터는 <script> 태그 내 변수로 임베드
  (나중에 데이터만 교체하면 앱이 업데이트되는 구조)

## 디자인
- 깔끔한 카드 기반 UI
- 과목별 색상 구분 (과목1: 파랑, 과목2: 초록, 과목3: 보라, 과목4: 주황)
- 정답: 초록 하이라이트, 오답: 빨강 하이라이트
- 진행률 바: 그라데이션
- 폰트: 시스템 폰트 사용

## 프롬프트 4: 법·제도 파트 전용 (웹 검색 병행)

```
당신은 빅데이터분석기사 필기시험 학습 콘텐츠 전문가입니다.
이번 세션에서는 법·제도 관련 파트의 JSON 데이터를 생성합니다.

이 파트는 법령 개정에 따라 내용이 변할 수 있으므로,
반드시 아래 공식 사이트를 우선 검색하여 최신 내용을 확인한 뒤 작성하라.

## 검색 우선순위 (공식 사이트 한정)
1순위: 개인정보보호위원회 (pipc.go.kr)
2순위: 법제처 국가법령정보센터 (law.go.kr)
3순위: 한국데이터산업진흥원 DataQ (dataq.or.kr)
4순위: EU 공식 GDPR 자료 (eur-lex.europa.eu / ec.europa.eu)

## 검색 필요 항목
1. 개인정보보호법 현행 주요 내용 및 최근 개정사항 (시행일 포함)
2. 데이터 3법 현행 구조 (개인정보보호법 중심 일원화 체계)
3. 가명정보 결합 절차 (결합전문기관, 반출심사위원회)
4. 개인정보 전송 요구권 시행 현황
5. 최근 개정 사항: 징벌적 과징금, CEO/CPO 책임, ISMS-P 관련 제도 변화 (연도 고정하지 말고 검색 결과 기준으로 채울 것)
6. 마이데이터 현황
7. GDPR 주요 원칙

⚠️ 중요: 검색으로 확인되지 않은 내용은 단정하지 마라.
확인 불가 시 verificationStatus: "review_needed"로 설정하고
content에 "※ 최신 시행 여부 확인 필요"를 명시하라.

## 서술 원칙
- "데이터 3법"은 2020년 개정의 역사적 배경으로 설명
- 현행 체계는 개인정보보호법 중심으로 서술
- 정보통신망법을 개인정보 보호의 독립 축으로 설명하지 않을 것
- 법령 조문 번호는 넣지 않되, 핵심 제도명과 방향은 정확하게
- 모든 항목에 sourceType: "law" 설정
- verificationStatus는 검색 확인 결과에 따라 "law_check" 또는 "review_needed" 설정

## 이번 세션 생성 수량

- 퀴즈: 정확히 15개

**파트1: 빅데이터의 이해** (개념 최소 12개 / 퀴즈 정확히 25개)

- 빅데이터 기술 및 제도: 빅데이터 플랫폼, 빅데이터와 인공지능, 개인정보 법·제도, 개인정보 활용

- 분석 작업 계획: 데이터 확보 계획, 분석 절차 및 작업 계획

**파트3: 데이터 수집 및 저장 계획** (개념 최소 10개 / 퀴즈 정확히 25개)
